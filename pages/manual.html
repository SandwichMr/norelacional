Proposito del Sistema

Biblioteca Juvenil es un sistema web basico para administrar una biblioteca escolar pequena. El sistema permite registrar autores, libros, usuarios y prestamos. Tambien permite consultar informacion en tablas, buscar registros, editar datos, actualizar documentos, eliminar registros e imprimir listas.

El objetivo es tener una herramienta sencilla para controlar que libros existen, quienes son sus autores, que alumnos usan la biblioteca y que prestamos estan activos.

Tecnologias Utilizadas

El proyecto usa tecnologias basicas de desarrollo web y base de datos no relacional:

  • HTML: estructura de las paginas.
  • CSS: estilos personalizados del sistema.
  • Bootstrap local: diseno de botones, tablas, tarjetas y formularios.
  • Bootstrap Icons local: iconos en menu, botones y titulos.
  • JavaScript basico: peticiones al servidor, busquedas e impresion.
  • SweetAlert2 local: mensajes de exito, error y confirmacion.
  • Node.js: servidor principal del proyecto.
  • Express: rutas con app.get y app.post.
  • MongoDB: base de datos no relacional.

Como Ejecutar el Sistema

Para ejecutar el sistema se debe tener MongoDB activo y despues correr el servidor Node desde la carpeta del proyecto.

cd /home/leonardo/biblioteca_juvenil
node server.js

Despues se abre el navegador en la siguiente direccion:

http://localhost:3000

El archivo principal del servidor es server.js. En ese archivo estan las rutas app.get para mostrar paginas y las rutas app.post para guardar, actualizar y eliminar datos.

Configuracion Necesaria

  • Servidor: Node.js ejecutando server.js en el puerto 3000.
  • Base de datos: MongoDB en mongodb://127.0.0.1:27017.
  • Nombre de la base: biblioteca_juvenil.
  • Archivos HTML: carpeta pages.
  • Archivos publicos: carpeta public.
  • Estilos: public/css/estilos.css.
  • JavaScript: public/js/script.js.
  • Imagenes: public/img.

Base de Datos y Colecciones

La base de datos se llama biblioteca_juvenil. Contiene cuatro colecciones principales:

  • autores: _id, nombre, nacionalidad, correo.
  • libros: _id, titulo, genero, anio, autorId.
  • usuarios: _id, nombre, telefono, correo, grupo.
  • prestamos: _id, libroId, usuarioId, fechaPrestamo, fechaEntrega, estado.

Relaciones NoSQL

MongoDB crea un campo _id para cada documento. El sistema usa ObjectId para guardar referencias entre colecciones.

En la coleccion libros, el campo autorId apunta al autor del libro. En la coleccion prestamos, los campos libroId y usuarioId apuntan al libro prestado y al usuario que lo pidio.

En la lista de prestamos se usa aggregate(). Dentro del aggregate se usa $lookup para relacionar prestamos con libros y usuarios. Tambien se usa $unwind para descomponer los arreglos relacionados y $project para mostrar solo los datos importantes.

prestamos
  $lookup con libros
  $lookup con usuarios
  $unwind libro
  $unwind usuario
  $project nombreUsuario, tituloLibro, fechas y estado

Funcionamiento de los Estilos

El diseno usa colores relacionados con una biblioteca: azul oscuro, blanco, beige, cafe claro y detalles dorados. La barra de navegacion mantiene el mismo estilo en todas las paginas.

Las tarjetas se usan en la pagina principal. Los formularios estan centrados y tienen fondo blanco. Las tablas tienen encabezados azul oscuro para que se lean con claridad. Los botones usan iconos para identificar acciones como guardar, editar, eliminar e imprimir.

Uso del Sistema Paso a Paso

  1. Entrar a http://localhost:3000.
  2. Usar la barra de navegacion para entrar a autores, libros, usuarios o prestamos.
  3. Entrar a una pagina de registro para llenar un formulario.
  4. Presionar el boton Guardar para mandar los datos al servidor.
  5. Entrar a la pagina de lista para ver los datos guardados.
  6. Usar el campo de busqueda para filtrar registros.
  7. Usar el boton de lapiz para editar un registro.
  8. Usar el boton de basura para eliminar un registro.
  9. Usar el boton de impresora para imprimir la tabla.

CRUD Paso a Paso

Crear: el usuario llena un formulario y presiona Guardar. El navegador manda los datos al servidor con fetch. El servidor recibe la informacion en una ruta app.post y usa insertOne para guardar en MongoDB.

Mostrar: el usuario entra a una lista. JavaScript pide datos a una ruta app.get. El servidor consulta MongoDB con find o aggregate y regresa los datos en formato JSON.

Editar: el usuario presiona el boton del lapiz. La pagina abre el formulario con el id del registro. Se carga la informacion con una ruta app.get y se llenan los campos.

Actualizar: al guardar cambios, el servidor usa updateOne con el ObjectId del documento.

Eliminar: al presionar el boton de basura, SweetAlert2 pide confirmacion. Si el usuario acepta, el servidor usa deleteOne.

Restricciones de Integridad

El sistema usa referencias por ObjectId. Para registrar un libro se debe seleccionar un autor existente. Para registrar un prestamo se debe seleccionar un libro existente y un usuario existente.

Como el sistema es basico y escolar, no usa seguridad avanzada ni reglas complejas. No se aplica cascade automatico. Si se elimina un autor, libro o usuario relacionado, el sistema no borra automaticamente otros documentos. La integridad se maneja de forma sencilla usando listas desplegables y referencias.

En este proyecto no se usa set null ni restring automatico de base de datos, porque MongoDB no maneja llaves foraneas como SQL. La restriccion principal es que los formularios obligan a seleccionar documentos existentes.

Validaciones y Manejo de Errores

  • Los campos importantes tienen required para evitar datos vacios.
  • Los correos usan type="email".
  • El telefono usa pattern y maxlength para capturar 10 numeros.
  • Las fechas de prestamos usan campos type="date".
  • SweetAlert2 muestra mensajes de exito al guardar, actualizar o eliminar.
  • SweetAlert2 muestra confirmacion antes de eliminar un registro.
  • Si ocurre un error en MongoDB, el servidor responde un mensaje de error y la pagina lo muestra.
